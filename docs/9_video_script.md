## Section 1 — Hook (0:00–0:45)

TCP and UDP are the two transport protocols that the entire internet is built on.
Every web request, every video stream, every DNS lookup, every online game — all
of it runs on one of these two protocols. But despite how fundamental they are,
most networking courses treat them as abstract concepts.

In this project, I built a custom measurement platform to answer a concrete
question: **what actually happens to TCP and UDP performance when you change
the size of your messages, or the size of your socket buffers?** How do both
protocols hold up under realistic network stress — congestion, high latency,
bufferbloat, and packet loss?

This is not a simulation. Every number in this presentation comes from a real
experiment running on a real machine, with real kernel socket calls, real
network emulation, and real measurements.

---

## Section 2 — Project Overview (0:45–1:30)

The project measures two independent variables. The first is **payload size** —
how large each message is — ranging from 64 bytes up to 64 kilobytes for TCP,
and up to 16 kilobytes for UDP. The second is **socket buffer size** — the
kernel memory allocated for SO_SNDBUF and SO_RCVBUF — ranging from 4 kilobytes
to 256 kilobytes.

Both variables are measured across **five network conditions**: a clean baseline,
a congested condition with a competing TCP flood, a high-latency condition with
50 milliseconds of one-way delay, a bufferbloat condition with a 1 megabit per
second link and a large queue, and a lossy condition with 5 percent random packet
drop.

The result is 230 TCP data rows, 211 UDP data rows, and **7 plots** — all
generated by a fully automated analysis pipeline. There are 40 automated tests
confirming every part of the measurement code is correct.

---

## Section 3 — Architecture and Tools (1:30–2:30)

Every experiment follows the same pattern. A **receiver thread** starts in the
background and binds to a port. A **sender** runs in the main thread, transmits
the configured number of messages, collects metrics, and appends one row to a
CSV file. Three runs per condition cell give us a mean and standard deviation.

The tools were chosen deliberately. **Python's socket library** gives direct
access to TCP and UDP sockets with no abstraction layer — meaning I can call
`setsockopt` directly to set SO_SNDBUF and SO_RCVBUF to any value I want and
observe the exact effect. **dummynet via pfctl** is the macOS equivalent of
Linux's tc/netem — it applies delay, bandwidth caps, queue limits, and packet
loss directly to the loopback interface at the kernel level. **pandas and
matplotlib** handle aggregation and plotting. **pytest** runs 40 automated tests
using a RESULTS_PATH patching pattern that redirects CSV output to a temp file
so experiments never touch real data.

The data is collected by running five scripts in order, taking roughly 70 to 85
minutes total, then running analyze.py to produce all seven plots.

## Section 4 — Results: Payload Sweep (2:30–6:00)

### Plot 1 — Throughput vs Payload Size

*[Show Plot 1]*

This is the baseline result with no network emulation. The blue line is TCP.
The orange line is UDP.

TCP scales from **82 megabits per second at 64 bytes** all the way to
**9,300 megabits per second at 64 kilobytes** — a 128 times increase. Every
TCP message requires the same sequence of work regardless of size: a system
call, a kernel context switch, ACK processing. At 64 bytes, that overhead
dominates — you are paying full cost for almost no data. At 64 kilobytes,
that same overhead is divided across a thousand times more bytes, making each
byte progressively cheaper. TCP also keeps the pipe continuously filled by
probing and increasing its congestion window, which amplifies this further.

UDP stays flat. It is rate-limited to 500 packets per second, so throughput is
simply rate times payload size. It grows linearly but stays low — not because
UDP is slow, but because the application is in control of the send rate. That
is a design choice, not a limitation.

### Plot 2 — TCP Latency vs Payload Size

*[Show Plot 2]*

This is the most important plot in the dataset. It shows round-trip time for
TCP under all five conditions simultaneously.

Baseline is near zero — loopback is memory, not a real network. The high
latency line sits at a stable **103 milliseconds** across every payload size —
exactly 2 times the 50 millisecond one-way delay that dummynet was configured
with. That flat line confirms the delay emulation is working correctly.

Now look at the bufferbloat line. At small payloads it is low. At 4 kilobytes
it climbs to 68 milliseconds. At **16 kilobytes it explodes to 1,060
milliseconds** — with a standard deviation of 740 milliseconds. One run
measured 20 milliseconds. Another run on the same machine, same link, same
experiment measured 1,604 milliseconds.

This is the textbook definition of **bufferbloat**. The 1 megabit per second
link is slow enough that each 16 kilobyte message takes 130 milliseconds to
drain. Even a few messages queued ahead of yours means hundreds of milliseconds
of wait time. And because the queue oscillates between full and draining between
runs, the latency is completely unpredictable. That unpredictability is what
makes bufferbloat so damaging in practice.

### Plot 3 — UDP Packet Loss vs Payload Size

*[Show Plot 3]*

Under the lossy condition, UDP loss holds near **4.9 percent** across every
payload size. dummynet drops packets probabilistically — it flips a coin per
packet, not per byte — so whether the datagram carries 64 bytes or 16 kilobytes,
it faces the same 5 percent drop chance.

The congested line sits at exactly zero. The background TCP flood does not cause
UDP datagram drops at 200 packets per second. This is an important finding:
UDP coexists with a competing TCP flow on loopback without any loss, because
the combined load never exhausts the queue.

### Plot 4 — UDP Jitter vs Payload Size

*[Show Plot 4]*

Jitter measures how irregular the inter-arrival gaps are between datagrams.

The counterintuitive result here is the **bufferbloat line**. It starts high at
small payloads — around 1.2 milliseconds — and drops toward zero at 16
kilobytes. When the 1 megabit per second link is fully saturated, the queue
never drains between packets. Packets exit the queue at a perfectly steady
1 megabit rate — the queue itself becomes the pacer. The result is low jitter
despite over a second of absolute latency. **Large queues regularize timing at
the cost of enormous delay** — that is a known, well-studied property of
buffered networks.

### Plot 5 — TCP vs UDP Under Congestion

*[Show Plot 5]*

This bar chart compares TCP and UDP throughput under the congested condition.
TCP fills the bars at every payload size. UDP is barely visible at the base.

I want to call out a known anomaly here. At 16 kilobytes, **congested TCP
achieves 11,579 megabits per second — higher than baseline's 7,887**. AIMD
theory predicts congestion should reduce throughput, not increase it. On
loopback, the RTT is 0.04 milliseconds. TCP detects a loss, halves its window,
and recovers to full speed within a single millisecond. The background flood
also keeps the kernel TCP fast path warm, which lowers effective RTT below
baseline. This is loopback-specific. On a real network with even 5 milliseconds
of RTT, the AIMD backoff penalty would persist long enough to show a clear and
sustained drop. The anomaly is acknowledged and documented in the report.

## Section 5 — Results: Buffer Sweep (6:00–7:30)

### Plot 6 — Throughput vs Socket Buffer Size

*[Show Plot 6]*

This plot answers the second half of the research question — what does socket
buffer size do to performance? TCP throughput ranges from **1,063 megabits per
second at 4 kilobytes to 1,334 at 128 kilobytes** — only a 25 percent swing
with large overlapping error bars.

The expected collapse at 4 kilobytes did not appear, and the reason is
fundamental. Buffer effects on throughput are governed by the bandwidth-delay
product: the amount of data that must be in-flight to keep a link fully
utilized. On loopback, the RTT is 0.09 milliseconds. At 1 gigabit per second,
the entire bandwidth-delay product is only about **11 kilobytes**. Even a 4
kilobyte buffer — which macOS doubles to 8 kilobytes — is already large enough
to cover that window. There is almost no data that needs to stay in-flight, so
the buffer is never the bottleneck.

On a real WAN link at 50 milliseconds RTT, that same 4 kilobyte buffer would
cap TCP throughput at roughly **328 kilobits per second** regardless of the
link speed. A 256 kilobyte buffer would be near-full throughput. The effect is
real and critical in production — loopback simply cannot demonstrate its
magnitude.

UDP is perfectly flat at 3.2 megabits per second. The rate-pacing loop
controls UDP throughput. SO_RCVBUF has no effect.

### Plot 7 — UDP One-Way Delay vs Socket Buffer Size

*[Show Plot 7]*

Under bufferbloat, UDP OWD is flat at around **516 milliseconds across all five
buffer sizes**. Increasing SO_RCVBUF from 4 kilobytes to 256 kilobytes has no
measurable effect on delay.

The reason is architectural. dummynet intercepts packets at the kernel packet
filter — before they reach the socket receive buffer. All queuing happens inside
dummynet's internal queue, and by the time packets exit the pipe and reach the
socket buffer they are already spaced at the 1 megabit per second drain rate.
The socket buffer sees a steady, slow stream it can absorb regardless of size.

This is a valid negative result. It demonstrates that socket-layer and
network-layer bufferbloat are distinct phenomena that require different
experimental setups to isolate.

## Section 6 — Key Takeaways (7:30–8:45)

Four findings I want to leave you with.

First: **payload size is the dominant driver of TCP throughput**, with a 128
times range from small to large messages. The mechanism is pure amortization
of per-message overhead — a concept that applies to every system that has fixed
per-operation costs.

Second: **bufferbloat is uniquely damaging because it is unpredictable**. A
stable 103 millisecond delay is something applications can buffer around. A
latency that varies between 20 and 1,600 milliseconds on the same connection
in the same second is not. This is why Active Queue Management algorithms like
CoDel exist.

Third: **UDP is a diagnostic instrument**. Because it has no reliability layer,
every measurement — loss rate, jitter, one-way delay — is an unfiltered signal
of what the network is actually doing. TCP hides the same problems in latency
variance and retransmit overhead.

Fourth: **buffer settings matter at scale, but not on loopback**. The 25 percent
TCP throughput variation here would be a hundred-fold variation on a real WAN
link. Socket buffer tuning is a major operational concern at cloud providers
precisely because the effect amplifies with RTT.

## Section 7 — Limitations and Close (8:45–10:00)

Two results in this project are anomalies that need honest acknowledgment.
Congested TCP outperforming baseline is a loopback artifact — AIMD recovery
takes microseconds at near-zero RTT. The subtle buffer sweep effect is also a
loopback artifact — the bandwidth-delay product is too small for buffer size to
matter. Both are documented in the report with full mechanistic explanations
and future work suggestions.

If I were to extend this project, the two highest-value additions would be
re-running the buffer sweep with 50 milliseconds of dummynet delay — which
would reveal the full throughput collapse at small buffers — and adding a
TCP_NODELAY comparison at small payloads, which would show how Nagle's algorithm
coalesces small messages and why real-time applications always disable it.

The platform is fully extensible. Adding a new condition means writing one
shell script and one plot function. Everything else — the CSV schema, the
aggregation logic, the test pattern — carries forward unchanged.

Thank you.
