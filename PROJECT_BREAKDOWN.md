## Step 2 — TCP Measurement Module (`src/tcp_module.py`)

### What it does

Runs a TCP server and client on localhost. The client sends a configurable number of messages at a configurable payload size, and both sides record timing and throughput. Results are appended to `results/tcp_results.csv`.

### Design

- The **server** runs in a background thread, accepts a single connection, reads until it has received the expected total bytes, then records elapsed time.
- The **client** connects, sends N messages of size `payload_bytes`, then closes.
- One experiment = one (payload_size, buffer_size, condition_label) triple.
- Throughput = total bytes transferred / elapsed time.
- RTT latency = measured per-message using a ping-pong mode (send one → wait for echo → record round-trip time), then averaged.

### Key socket options to vary

| Option | Purpose |
|--------|---------|
| `socket.SO_SNDBUF` | Send buffer size — artificially lowering this creates backpressure |
| `socket.SO_RCVBUF` | Receive buffer size — inflating this simulates bufferbloat |
| `socket.TCP_NODELAY` | Disables Nagle's algorithm — relevant for small packet tests |

### What to implement

**`run_tcp_throughput(payload_size, buffer_size, n_messages, label)`**
- Spawns a server thread, runs client, measures total throughput
- Returns `(throughput_mbps, avg_latency_ms)`

**`run_tcp_latency(payload_size, buffer_size, n_pings, label)`**
- Ping-pong mode: client sends, server echoes, client records RTT
- Returns list of RTT samples (you'll compute mean/stdev in the pipeline)

**`save_result(row, filepath)`**
- Appends a dict as a CSV row

### CSV schema

```
protocol, payload_bytes, buffer_bytes, condition, throughput_mbps, avg_latency_ms, latency_stdev_ms, run_index
```

## Step 3 — UDP Measurement Module (`src/udp_module.py`)

### What it does

Same structure as the TCP module, but UDP. Because UDP is connectionless and unreliable, you also track **packet loss** and **jitter**.

### Design

- The **server** binds on a port and listens. Each message it receives is stamped with a sequence number sent by the client. After a timeout with no new packets, it stops and reports how many sequence numbers are missing (packet loss).
- The **client** sends N datagrams of `payload_bytes` at a fixed rate, each prefixed with an 8-byte sequence number and 8-byte send timestamp (use `struct.pack`).
- Jitter = mean absolute difference between consecutive inter-arrival times at the receiver (RFC 3550 definition).

### Key differences from TCP module

| Concern | Approach |
|---------|---------|
| No connection state | Server uses `recvfrom` in a loop with a timeout |
| No delivery guarantee | Track sequence numbers to compute loss rate |
| No flow control | Add a small `time.sleep` between sends to avoid flooding loopback |
| Message boundaries preserved | Each `sendto` is one datagram — no reassembly needed |

### What to implement

**`run_udp_experiment(payload_size, buffer_size, n_messages, send_rate_pps, label)`**
- Spawns a receiver thread, runs sender, collects results
- Returns `(throughput_mbps, loss_rate_pct, jitter_ms)`

**`save_result(row, filepath)`**
- Same CSV writer as TCP (you can share a utility function)

### CSV schema

```
protocol, payload_bytes, buffer_bytes, condition, throughput_mbps, loss_rate_pct, jitter_ms, run_index
```

### Note on large UDP payloads

UDP datagrams with payload > ~1472 bytes will be **IP-fragmented** on a typical Ethernet MTU (1500 bytes). On loopback (MTU 16384), fragmentation starts above ~16376 bytes. Testing payloads above 16KB on loopback will trigger actual IP fragmentation and reassembly — this is a real observable effect worth noting in your report.

## Step 4 — Network Emulation (`src/emulate.sh`)

### Why dummynet instead of tc/netem

macOS doesn't ship with the Linux traffic control stack. Instead, it has **dummynet**, a BSD packet scheduler accessible via `pfctl`. It supports:
- Bandwidth limits (`bw`)
- Propagation delay (`delay`)
- Packet loss rate (`plr`)
- Queue size limits (`queue`) — this is what simulates bufferbloat

### Core concepts

```
pipe → a virtual link with configurable bandwidth, delay, queue, loss
rule → a pf rule that steers traffic into a pipe
```

### Script structure

The script should expose four functions (call via `bash emulate.sh <command>`):

**`setup_pipe <delay_ms> <bw_mbit> <queue_slots> <loss_rate>`**
```bash
# Example: 20ms delay, 10Mbit/s, 100-slot queue, 1% loss
sudo dnctl pipe 1 config delay 20 bw 10Mbit/s queue 100 plr 0.01
# Route loopback traffic through it
echo "dummynet out quick on lo0 all pipe 1" | sudo pfctl -f -
sudo pfctl -e
```

**`teardown`**
```bash
sudo dnctl -q flush       # remove all pipes
sudo pfctl -f /etc/pf.conf  # restore default rules
sudo pfctl -d             # disable pf (re-enable default passthrough)
```

**`status`**
```bash
sudo dnctl show           # prints pipe stats: packets, bytes, queue depth
sudo pfctl -s rules
```

**`bufferbloat_scenario`**
- Sets a very large queue (e.g., 1000 slots) with limited bandwidth to induce queuing delay
- This is the key condition for demonstrating bufferbloat

### Usage pattern in experiments

```bash
bash src/emulate.sh setup_pipe 0 100 32 0.0   # baseline: no delay, high bw, small queue
# ... run experiment ...
bash src/emulate.sh setup_pipe 20 1 1000 0.0  # bufferbloat: slow link + huge queue
# ... run experiment ...
bash src/emulate.sh teardown
```

### Conditions to define

| Label | delay | bw | queue | plr | Purpose |
|-------|-------|----|-------|-----|---------|
| `baseline` | 0ms | 100Mbit | 32 | 0 | No emulation |
| `high_latency` | 50ms | 100Mbit | 32 | 0 | Propagation delay effect |
| `bufferbloat` | 0ms | 1Mbit | 1000 | 0 | Large queue + slow link |
| `lossy` | 0ms | 100Mbit | 32 | 0.05 | 5% loss |
| `congested` | 20ms | 2Mbit | 500 | 0.01 | Combined stress |

## Step 5 — Data Pipeline (`src/analyze.py`)

### What it does

Loads both CSVs, computes derived metrics, and prepares structured data for plotting. This is the glue between raw measurements and the final figures.

### Key operations

**Loading**
```python
import pandas as pd
tcp_df = pd.read_csv('results/tcp_results.csv')
udp_df = pd.read_csv('results/udp_results.csv')
```

**Aggregation across runs**
- Group by `(payload_bytes, buffer_bytes, condition)` and compute mean + 95% CI across the 3 run indices.
- `pandas.DataFrame.groupby(...).agg(['mean', 'std'])` handles this cleanly.

**Derived metrics**
- **Goodput efficiency** = `throughput_mbps / theoretical_max` — useful for normalizing across conditions
- **Overhead ratio** = `(payload_bytes + header_bytes) / payload_bytes` — TCP header is 20–60 bytes, UDP is 8 bytes; at small payloads this ratio is significant
- **Jitter-to-latency ratio** (UDP only) — measure of stability

### Output

A set of clean DataFrames, one per experiment dimension, that the visualization functions consume directly. Keep pipeline logic and plotting logic in separate functions — it makes iteration easier.

## Step 7 — Experiment Orchestration (`run_experiments.sh`)

This script ties everything together. It loops over all conditions and payload sizes, calling the Python modules and emulation script in sequence, and runs each condition 3 times.

### Structure

```bash
#!/usr/bin/env bash
set -e

PAYLOAD_SIZES=(64 256 1024 4096 16384 65536)   # bytes
BUFFER_SIZES=(4096 16384 65536 262144)          # socket buffer bytes
N_MESSAGES=1000
N_RUNS=3

run_condition() {
    local label=$1
    local delay=$2 bw=$3 queue=$4 plr=$5

    bash src/emulate.sh setup_pipe $delay $bw $queue $plr

    for run in $(seq 1 $N_RUNS); do
        for payload in "${PAYLOAD_SIZES[@]}"; do
            for buf in "${BUFFER_SIZES[@]}"; do
                python3 src/tcp_module.py --payload $payload --buffer $buf \
                    --messages $N_MESSAGES --label $label --run $run
                python3 src/udp_module.py --payload $payload --buffer $buf \
                    --messages $N_MESSAGES --label $label --run $run
            done
        done
    done

    bash src/emulate.sh teardown
}

run_condition baseline    0  100  32   0.0
run_condition high_latency 50 100 32  0.0
run_condition bufferbloat  0   1  1000 0.0
run_condition lossy        0  100  32  0.05
run_condition congested   20   2  500  0.01

python3 src/analyze.py   # generate all plots from aggregated CSVs
echo "Done. Plots saved to plots/"
```

### Estimated runtime

At 1000 messages × 6 payload sizes × 4 buffer sizes × 5 conditions × 3 runs × 2 protocols = 720 experiment cells. Keep individual experiments short (a few seconds each). Total wall time will likely be 20–40 minutes depending on the slow-link conditions.

## Step 8 — Report Notes

### Key arguments to make

1. **Fragmentation threshold**: Show the payload size at which performance degrades on loopback (~16KB) and explain it in terms of IP fragmentation + reassembly cost.

2. **Bufferbloat signature**: The bufferbloat condition should produce low packet loss but high latency variance — this is the defining characteristic. Plot 2 should show this clearly.

3. **TCP vs UDP tradeoff**: TCP self-throttles under congestion (lower throughput, stable latency); UDP does not (higher throughput, rising loss). Neither is universally better — application requirements determine the choice.

4. **Header overhead at small payloads**: TCP has 20–60 bytes of headers vs UDP's 8 bytes. At 64-byte payloads, TCP overhead is proportionally significant. Show this with the overhead ratio metric.

### Data to include in the report

- All 5 plots
- Table of raw mean throughputs per condition
- At least one concrete example of the bufferbloat effect with numbers (e.g., "under the bufferbloat condition, 64KB payloads experienced 3× higher latency than baseline despite identical loss rates")
